# 프로젝트 제약과 맥락

> 2026-04-21 확정. 엔진/맵 리팩토링의 모든 판단은 이 전제 위에서 이루어집니다.

## 규모

- **에이전트 수 상한: 약 100개**. 이 규모에서 매끄럽게 동작하는 구성을 기준으로 삼습니다. 성능 최적화는 이 선에서 충분하면 됩니다.

## 배포 타깃

- **현재**: 웹 브라우저 (VS Code 사이드 패널 임베드 포함)
- **가능한 미래**: Tauri 또는 Electron 기반 데스크톱 앱으로 마이그레이션 가능성 있음
- **의미**: React 친화성은 계속 중요. 번들 크기는 당장은 중요하지만 중장기로는 완화됨. WASM 40MB 급 엔진(Godot)은 웹 단계에서 이미 탈락.

## 맵 저작/배포 모델

- **디자이너가 Tiled로 맵을 만들고 제품에 함께 번들링해 배포**.
- **런타임 맵 교체·편집 없음**. 사용자(최종 사용자)는 맵을 바꿀 수 없음.
- **예외로 허용**: 여러 벌의 tile set을 동봉하고 사용자가 그중 하나를 **런타임에 선택**하는 UI는 허용.
- **asset/구성 교체 경로**: 프로젝트 fork 후 개발자가 수동 교체. 동적 asset 교체 시스템은 현재 범위 밖.
- **의미**: 인게임 맵 에디터가 필요 없음. 기존 `src/office/editor/`는 Tiled 전환 이후 역할이 없으므로 제거/재작성 대상.

## 마이그레이션 방침

- **전면 교체 / 전체 재작성 허용**. 기존 코드 보존이 제약이 아님.
- 따라서 엔진 선택 시 "기존 로직과의 공존 가능성"의 가중치가 크게 떨어지고, "쯔꾸루식 결과물을 빠르게 뽑는 능력"과 "Tiled 워크플로우 1급 지원"의 가중치가 올라감.
- 단, **보존해야 하는 것은 "코드"가 아니라 "에이전트 상태 → 시각 상태 매핑 개념"**. 걷기·타이핑·좌석·사회적 버블 같은 observability semantic은 새 엔진 위에서도 동일하게 표현되어야 함.
